Chapter 2

Why do LLMs hallucinate?

An important insight here is that LLMs do not think or reason before they write, the writing IS the reasoning. This means that forcing an LLM to use fewer words can hurt its performance, and conversely letting it write more can improve its performance.

The problem, same both times
What's 842,019 × 109,310?
Just asked
(no additional instructions)
92,025,517,890
Wrong.
Asked to reason
"…Reason it through."
Let me break this into parts:
Split 109,310 into 100,000 + 9,000 + 300 + 10
842,019 × 100,000 = 84,201,900,000
842,019 × 9,000 = 7,578,171,000
842,019 × 300 = 252,605,700
842,019 × 10 = 8,420,190
Adding:
92,041,096,890.
Correct.

Sonnet 4.6

  • Math is normally trivial for computers, but LLMs can run into issues. For simple math problems using common numbers, LLMs will usually get the answer right. Those problems are part of the LLMs' training data, so the model will be strongly pointed towards the right answer.
  • LLMs are designed to give answers that look correct. For uncommon equations like the one above, the LLM is just doing its best to get to an answer that seems plausible. This is what's scary about AI hallucinations. The answers they give seem correct, because they've been heavily trained to look correct!
  • More tokens allow for more reasoning. You may have seen some LLMs with a thinking section before they give out their final output. As seen above, giving the LLM additional tokens to reason with can improve performance. Those thinking tokens are not a special internal part of the AI, they're exactly the same as the tokens in the final output. Other models may still be doing this "thinking" but not showing it to you in the output.
  • Current AIs often solve problems like these by calling a tool. When I tried the above equation on Opus 4.7, it chose to build a quick python script to get the answer rather than trying to reason it out.


LLMs function by providing a plausible next token, but looking plausible is not the same as being true. Take the case of Roberto Mata v. Avianca, Inc., in which a lawyer generated false cases.

Lawyer asks ChatGPT
Find cases where a statute of limitations was tolled for a Montreal Convention claim.
ChatGPT returns
Varghese v. China Southern Airlines Co., Ltd.,
925 F.3d 1339 (11th Cir. 2019).
Holding that the Montreal Convention's two-year limitations period was tolled during the pendency of a related bankruptcy proceeding...

None of this exists. The case was entirely made up by ChatGPT to sound like something that could exist, without actually checking for a source. The lawyers that were responsible for submitting this case were fined $5,000 and ordered to notify each judge that they improperly cited.

Chapter 3

So why does this actually work?

If an LLM is "just" predicting the next token without any underlying understanding, why is it so accurate so often?

The short answer is that LLMs have a LOT of training data, and that "plausible" has a lot of overlap with "true". LLMs are trained by looking at pre-existing data, and using that data to decide how to continue their writing. That data is part of what determines the probabilities we saw in the previous sections. When asked to answer a math question, the LLM has tons of math data that it looks at to point it in the right direction. When asked to find a specific case, it has huge amounts of legal data that it can look at in order to find what you're looking for.

A few things worth noticing:

  • "Jurisprudence" splits into multiple tokens. Tokens are not always discrete words, uncommon words tend to use multiple tokens. There can also be differences in token usage depending on capitalization. In the input (lowercase, with a leading space) it becomes three tokens: ·juris, pr, udence. In the output (capital J, no leading space, sentence-start) it becomes four: Jur, is, pr, udence. For simplicity, we'll be using mostly single word tokens.
  • Tokens reuse numeric IDs across contexts. pr is token 652 both times it appears. udence is 74193. ·philosophy (with leading space) is 19675. These assignments are made when the LLM is trained, they don't change during usage.
  • The model never actually sees English. Only numbers. Whatever it "knows" about grammar, meaning, or legal theory is computed from operations on these token IDs.
  • Tokens determine cost. The amount that you're charged for using an LLM is based on both the number of tokens you input + the number of tokens returned.
Chapter 2

Embeddings

For every token, the LLM has a corresponding vector (a list of numbers). These are called the model's embeddings. These embeddings are created during the model's initial training, and don't change afterwards. Each number in the embedding means nothing by itself, but embeddings define the relationship that each token has with other tokens. Think of the embeddings as defining a point in space. Tokens that appear in similar contexts are usually found near each other. What's interesting is that we can find meaning by adding or subtracting the embeddings from each other. For example, if you take the embedding for King, subtract the embedding for Man, and add the embedding for Woman, you'll get close to the token for Queen. The example below represents tokens in two dimensions, but real LLMs have many many dimensions that define relationships.

A few things worth noticing:

  • Categories form visible clusters. Words that are related to each other in meaning appear near each other. Nothing in the training told the LLM that these were animals or colours, it read billions of words of text and through pattern matching, learned that some words belong near others.
  • This is a flattening of real embeddings. Real embeddings can be over fifty dimensions, this display is just a representation.
  • There is no meaning to the axes. A token being towards the top or bottom has no meaning, except in relation to the other tokens.

Transformers

The map above is static, each token always maps to one fixed embedding. But in real language, the meaning of a word is usually determined by its context. Take the word charge: it could be a legal charge, a credit-card charge, an electrical charge, or a cavalry charge. The token is always the same, and always points to the same embedding. So how does the model know which one you mean?

The answer is the transformer, the neural network architecture at the core of every modern LLM. The transformer takes the embeddings above, and modifies their positions based on context. This is why your LLM conversations can only get so large before the LLM is forced to forget things, every token in your context affects every new token. The transformer is what gives those probabilities that we saw at the beginning.

The mechanism that does this is called attention: for every token in a sentence, the model computes how much each other token matters. The demo below shows exactly that happening to the word charge across four different sentences.

This is the core of how a modern LLM works. The loop, tokenization → embedding → transformer → output, runs once for every token you type into an LLM, and once more for every token it writes back. A conversation is thousands of repetitions of that same small cycle.
